# 执行追踪和性能分析

## 概述

Buckyball 提供全面的执行追踪基础设施，用于调试、性能分析和设计验证。本指南涵盖追踪生成、追踪格式分析和常见分析工作流程。

## 追踪基础设施

### 追踪类型

Buckyball 在模拟期间生成多个追踪流，每个流捕获系统行为的特定方面。**ITRACE** 流捕获指令发放和完成事件及 RoB 条目标识符，非常适合调试指令序列和识别停顿。**MTRACE** 流记录内存操作、load/store 模式和 DMA 活动，用于分析缓存行为和内存瓶颈。**PMCTRACE** 流发出性能计数器事件，包括缓存命中和缺失，用于系统级分析和吞吐量测量。**CTRACE** 流记录提交和退役事件，用于验证程序正确性。**BANKTRACE** 流监控库访问模式和计分板状态，对寄存器压力分析和重命名效率评估有用。

### 启用追踪

追踪通过模拟期间的 `bdb_trace_mask` 环境变量控制。掩码中的每一位启用特定的追踪类型：第 0 位启用 ITRACE（0x01），第 1 位启用 MTRACE（0x02），第 2 位启用 PMCTRACE（0x04），第 3 位启用 CTRACE（0x08），第 4 位启用 BANKTRACE（0x10）。常见配置包括 0x01 仅用于指令调试，0x03 用于指令和内存分析，0x1F 同时启用所有追踪类型。追踪以 NDJSON 格式写入标准输出，其中每行表示单个事件。

## NDJSON 追踪格式

### 记录类型

#### 指令追踪（ITRACE）

指令追踪记录捕获指令通过发放流水线的生命周期。每条记录包括时钟周期时间戳、RoB 条目 ID、目标执行域（ball、mem、gp 等）、指令编码、Ball 操作的功能选择字段（funct7）和程序计数器。这些字段允许将高级程序行为与低级硬件事件关联。

#### 内存追踪（MTRACE）

内存追踪记录记录 load、store 和 DMA 操作。每条记录指定操作类型（load、store、dma_read 或 dma_write）、操作开始的时钟周期、虚拟或物理地址、传输的字节数、hart（核）ID、访问是否在缓存中命中以及操作持续的周期数。分析这些记录可以揭示访问模式、缓存效率，并识别内存子系统瓶颈。

#### 性能计数器追踪（PMCTRACE）

```json
{
  "kind": "pmctrace_cache",
  "clk": 100,
  "event": "miss",
  "cache_level": "l1",
  "hart": 0
}
```

#### 提交追踪（CTRACE）

```json
{
  "kind": "ctrace_commit",
  "clk": 200,
  "hart": 0,
  "pc": "0x80001a40",
  "instr": "0x002082b3"
}
```

## 追踪分析工具

### NDJSON 可视化脚本

`bdb_ndjson_viz.py` 脚本从追踪文件生成 RoB 活动的时间轴可视化。它显示每个 RoB 条目的生命周期（分配、发放和完成）、随时间的指令发放率、域利用率（Ball、内存和 GP 带宽的使用程度），以及揭示何时没有指令活跃的停顿模式。输出是 PNG 时间轴图像，使得视觉上很容易发现性能问题。

### 追踪注解

`bdb_ndjson_annotate.py` 脚本通过交叉引用指令编码与 ISA 定义和符号表来用符号信息充实追踪。它添加函数名（通过符号表查找）、指令助记符（从 ISA 定义解码）、美化打印的操作数和调用/返回标记。注解追踪更易读，并使得关联高级程序行为与低级执行模式变得更容易。

### 手动追踪处理

用 Python 解析 NDJSON 追踪：

```python
import json
from pathlib import Path

records = []
with open('trace.ndjson') as f:
  for line in f:
    records.append(json.loads(line))

# 筛选指令完成
completions = [r for r in records if r['kind'] == 'itrace_complete']

# 测量各域平均延迟
domain_latencies = {}
for r in completions:
  domain = r['domain_id']
  latency = r['clk']  # 简化；实际延迟需要发放到完成映射
  if domain not in domain_latencies:
    domain_latencies[domain] = []
  domain_latencies[domain].append(latency)

for domain, lats in domain_latencies.items():
  print(f"{domain}: avg {sum(lats)/len(lats):.1f} cycles")
```

## 常见分析工作流程

### 识别性能瓶颈

当应用程序吞吐量低于预期时，基于追踪的系统分析可以指出根本原因。首先启用所有追踪类型（BDB_TRACE_MASK=0x1F）并运行工作负载。可视化脚本将显示大的空闲间隙（没有活跃 RoB 条目的时期）指示停顿条件，或一致的低 RoB 占用率指示指令流本身是瓶颈。使用 grep 对追踪文件进行域特定分析揭示 Ball、内存或 GP 域是否饱和。检查 fence 和 barrier 事件显示同步开销是否显著。这种系统化方法快速缩小问题是在指令获取、特定执行域、内存子系统还是同步中。

### 调试不正确的结果

当工作负载产生错误的输出时，启用提交追踪（BDB_TRACE_MASK=0x08）记录已提交指令的序列。提取提交序列并使用 objdump 比较程序计数器值与预期程序流。如果特定指令或地址范围看起来可疑，检查该地址周围的内存追踪以查看是否发生了意外的内存操作。交叉引用指令追踪与提交追踪揭示指令是否被发放但从未提交，这通常表示缺少同步原语或不正确的 barrier/fence 放置。

### 分析内存访问模式

内存子系统优化始于为代表性工作负载捕获内存追踪（BDB_TRACE_MASK=0x02）。通过计数缓存命中与缺失和测量延迟分布来聚合访问统计以确定整体缓存有效性。通过查找最频繁访问的地址范围来识别内存热点，这通常揭示更好的数据布局或预取机会。高延迟访问加上缺失事件表明内存子系统拥塞或缓存利用率差，这可能需要增加缓存容量或调整关联度。

### 寄存器压力分析

寄存器压力可以通过首先捕获库追踪（BDB_TRACE_MASK=0x10）然后分析库利用率模式来优化。计数每个库的同时写入以确定任何单个库是否是瓶颈。通过检查别名重用与新分配的比率来测量重命名效率。高别名重用指示有效的寄存器分配，而频繁的库停顿表明虚拟库配置对于工作负载可能不足，在这种情况下增加虚拟库的数量可能改进性能。

## 追踪存储和管理

### 文件大小考量

追踪文件随模拟长度和启用的追踪类型快速增长。仅 ITRACE 每 1000 周期生成约 50-100 KB，而启用所有追踪类型可生成 200-500 KB 每 1000 周期。超过 100 万周期的大工作负载可能生成 100+ MB 追踪文件，这可能消耗大量磁盘空间并变得难以处理进行分析。规划追踪持续时间和仅选择必要的追踪类型有助于管理存储需求。

### 选择性追踪

通过使用环境变量限制追踪持续时间来减少存储需求和开销。BDB_TRACE_ADDR_START 和 BDB_TRACE_ADDR_END 变量将追踪限制到特定地址范围，在专注特定数据结构或内存区域时有用。BDB_TRACE_MAX_CYCLES 变量将追踪限制到从模拟开始固定数量的时钟周期，有助于隔离早期行为或稳态性能特征而无需追踪整个执行。

### 压缩

NDJSON 追踪文件可以使用 gzip 压缩以显著减少归档存储。压缩追踪保留完整信息，同时通常实现 5-10 倍的压缩比。分析工具可以直接通过标准 Unix 管道读取压缩追踪，所以在处理之前不需要重新压缩。

## 追踪收集故障排除

### 无追踪输出

如果模拟完成但未生成追踪文件，首先验证 BDB_TRACE_MASK 设置为非零值（echo $BDB_TRACE_MASK）。通过 grep 查找"trace"关键字来检查模拟日志中的追踪配置消息。确认追踪掩码至少设置了第 0-4 位之一，因为每一位启用特定的追踪类型。最后，确保追踪将被写入的目录可写且有足够的可用空间。

### 不完整追踪

如果追踪文件被截断或缺少事件，增加日志缓冲区大小（BDB_TRACE_BUFFER_SIZE=1000000）以允许模拟在刷新前排队更多事件。减少模拟步长会导致追踪更频繁地刷新，这可以帮助防止在长时间运行的测试期间丢失。检查可用磁盘空间，因为大工作负载可能填满驱动器并截断输出。如果一切都失败，运行更短的测试以较少的模拟周期来建立基线。

### 高追踪开销

启用追踪由于生成和写入事件的成本增加模拟运行时。为最小化开销，通过将 BDB_TRACE_MASK 减少到仅分析所需的类型来禁用未使用的追踪类型。使用窗口（BDB_TRACE_ADDR_START/END 或 BDB_TRACE_MAX_CYCLES）仅追踪感兴趣的区域而不是整个执行。运行具有较少模拟周期的较短测试以进行分析，并考虑在更快的机器上运行分析，因为 Verilator 吞吐量随 CPU 速度扩展。
